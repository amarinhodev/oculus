# OCULUS Analyzer Workflow (V9.2 — RAG-Native Vault Discovery)

> **Core principle:** This skill contains NO hardcoded knowledge about projects,
> systems, or people. It DISCOVERS the user's universe via semantic queries to the RAG
> (`obsidian_rag_index`) before analyzing any transcription. This ensures it works
> for ANY user, with any vault, without manual file reading.

---

## 📋 Prerequisites

This skill depends on:
- **Gemini CLI** with the `gemini-obsidian` extension installed and configured
- **Obsidian Vault** indexed by the RAG (`obsidian_rag_index`)

If the RAG is unavailable, the skill still works but generates notes with less context
(without automatic WikiLinks for vault projects/people).

---

## Step 0 — Read Transcription Frontmatter

The `.md` file generated by `processor.py` contains rich YAML frontmatter. Read and use it as initial context:

```yaml
title: "Meeting title"
date: "YYYY-MM-DD"
time: "HH:MM"
type: "meeting | daily | weekly | review | ..."
participants: ["Name1", "Name2"]
tags: [tag1, tag2]
preferred_language: "Brazilian Portuguese | English | Spanish | ..."
```

**Action:** Extract and store in memory the fields `title`, `date`, `time`, `type`, `participants`, `tags`,
and `preferred_language` (the `PREFERRED_LANGUAGE` value passed by `watcher.py`).
Use `date` and `time` to name the output file and locate the correct daily log.

---

## Step 1 — Vault Discovery via RAG (MANDATORY — EXECUTE BEFORE ANALYSIS)

> **ABSOLUTE RULE:** Never ask the user anything. Never list directories manually.
> Never read frontmatter file by file. Use **exclusively** semantic queries to the RAG.

---

> **IMPORTANT — Query Language:**
> Execute all RAG queries (Steps 1.1, 1.2, 1.3 and Step 2) in the **SAME language**
> as `PREFERRED_LANGUAGE` passed in the prompt. The query language must match the
> vault content language to maximize semantic recall accuracy.
>
> Examples:
> - `PREFERRED_LANGUAGE = "Brazilian Portuguese"` → queries in Portuguese
> - `PREFERRED_LANGUAGE = "English"` → queries in English
> - `PREFERRED_LANGUAGE = "Spanish"` → queries in Spanish

---

### Step 1.1 — Active Projects Query

Execute the following query **in PREFERRED_LANGUAGE** via `obsidian_rag_index`:

> "What are the active or in-progress projects in this vault? List name, status and relative path."
> *(PT-BR example: "Quais são os projetos ativos ou em andamento neste vault? Liste nome, status e caminho relativo.")*

The RAG returns excerpts from project notes with rich context — name, status, path — without needing
to iterate files manually. Process the response to extract:
- Project name (title or alias)
- Status (active, in progress, paused, completed)
- Relative path in the vault (to generate correct WikiLinks)

### Step 1.2 — Recurring People Query

Execute the following query **in PREFERRED_LANGUAGE** via `obsidian_rag_index`:

> "Who are the people frequently mentioned in this vault? Identify names that appear in multiple notes."
> *(PT-BR example: "Quem são as pessoas mencionadas com frequência neste vault?")*

The RAG identifies people semantically, cross-referencing multiple notes. Process the response to extract:
- Full name or reference name
- Path of the personal note (if it exists in the vault)

### Step 1.3 — Systems and Tools Query

Execute the following query **in PREFERRED_LANGUAGE** via `obsidian_rag_index`:

> "What systems, tools, APIs or platforms are mentioned in this vault? List acronyms and full names."
> *(PT-BR example: "Quais sistemas, ferramentas, APIs ou plataformas são mencionados neste vault?")*

Captures technical context (ServiceNow, JOCA, AWS, Jira, Slack, etc.) without hardcoding. Process the response
to extract:
- System or tool name/acronym
- Usage context (brief description if available)

### Step 1.4 — Build Consolidated Context

From the responses of queries 1.1, 1.2, and 1.3, build the `vault_context` in memory:

```python
vault_context = {
    "projects": {
        "name_or_alias": {
            "path": "relative/path/in/vault.md",
            "status": "active|in_progress|paused|done",
        },
        # ...
    },
    "people": {
        "person_name": "path/to/person/note.md",  # or None if no dedicated note
        # ...
    },
    "systems": {
        "acronym_or_name": "brief_description",
        # ...
    },
}
```

This dictionary completely replaces the `project_universe` and `people_universe` from V9.0 and is used
in all subsequent steps for matching and WikiLink generation.

### Step 1.5 — Graceful Fallback

If the RAG is unavailable or returns an empty response for **any** of queries 1.1–1.3:

1. Log WARNING to console (do not abort execution):
   ```
   [OCULUS WARNING] obsidian_rag_index unavailable or no response. Continuing without vault_context.
   ```
2. Initialize empty `vault_context`: `{"projects": {}, "people": {}, "systems": {}}`
3. Continue to Step 2 — analysis proceeds normally
4. The generated note will have fewer automatic WikiLinks, but summary, decisions, and action items will work

> **Step 1 Result:** `vault_context` dictionary in memory with projects, people, and systems
> discovered via semantic queries to the RAG — zero manual file reading or directory listing.

## Step 2 — Semantic Analysis with Context

Analyze the **transcription content** using the `vault_context` discovered via RAG in Step 1.

> **IMPORTANT — Query Language:** Execute all RAG queries in this step in the **SAME language**
> as `PREFERRED_LANGUAGE`. This is critical for accurate semantic matching of entities, acronyms,
> and domain-specific terminology that appear in the transcription.

### 2.1 — Project and System Matching via RAG Query

For entities mentioned in the transcription, **use RAG query** instead of manual matching.
Formulate the query in PREFERRED_LANGUAGE:

1. Identify key entities mentioned in the transcription (projects, systems, acronyms, tools)
2. Execute query via `obsidian_rag_index` **in PREFERRED_LANGUAGE** with transcription context:
   ```
   "This meeting mentioned [X], [Y], [Z]. Which notes in the vault are related to these topics?"
   ```
   *(Adapt phrasing to PREFERRED_LANGUAGE; replace `[X]`, `[Y]`, `[Z]` with the identified entities — max. 5–8 per query.)*
3. The RAG returns paths of relevant notes with automatic semantic matching — without needing
   manual logic for abbreviations or synonyms (e.g., "Snow" → ServiceNow, "bot" → JOCA).
4. Record `matched_notes: [{"entity": "...", "note": "[[NoteName]]", "path": "..."}]`

If RAG is unavailable: use `vault_context` from Step 1 for simple string similarity matching (fallback).

### 2.2 — People Matching

1. For each name mentioned in the transcription, check for a match in `vault_context["people"]`
2. If found: record as `[[WikiLink]]`
3. If not found in `vault_context`: try specific RAG query **in PREFERRED_LANGUAGE**:
   ```
   "Is there a note in the vault for a person named [Name]?"
   ```
4. If not found in either: keep as plain text

### 2.3 — Structured Content Extraction

Extract from the transcription:
- **Decisions** — affirmative conclusive phrases ("it was decided", "we agreed", "approved")
- **Action Items** — commitments with a future or imperative verb + mentioned responsible party
- **Insights** — technical points, alerts, relevant discoveries mentioned

## Step 3 — Generate Meeting Note

Generate a Markdown file using the following template (fill with real data from the analysis):

```markdown
---
title: "<meeting title — from transcription frontmatter>"
date: "YYYY-MM-DD"
time: "HH:MM"
type: "<type — from transcription frontmatter>"
participants:
  - "<Participant1>"
  - "<Participant2>"
related_projects:
  - "[[ProjectName1]]"
  - "[[ProjectName2]]"
tags:
  - meeting
  - <type>
  - <tags detected from transcription>
---

# 🎙️ <meeting title>

## 📋 Executive Summary
<objective and main outcome in 2-3 lines — inferred from the transcription>

## ✅ Decisions Made
- <decision 1>
- <decision 2>

## 📌 Action Items
- [ ] <task> (@responsible) ← detect responsible mention in transcription; omit @responsible if not mentioned

## 💡 Technical / Business Insights
<key points from the conversation — technical, strategic, alerts>

## 🔗 Related Projects and Contexts
← WikiLinks generated dynamically from RAG queries (Step 2.1)
- [[ProjectName]] — <why it is relevant to this meeting>

## 👥 Participants
← Use [[WikiLink]] if the person has a note in the vault (vault_context["people"]), otherwise plain text
- [[PersonName]] or Person Name
```

**Generation rules:**
- Generate the note in the **language specified in the prompt** (passed by `watcher.py`); default: English
- Never invent projects or people — use only what was discovered via RAG or explicitly mentioned in the transcription
- If no project is identified in the semantic analysis, omit the `## 🔗 Projects` section
- `related_projects` in the frontmatter should contain only projects returned by the RAG with confirmed matching

---

## Step 4 — Daily Log Integration (GAMA Protocol)

> **ABSOLUTE RULE:** Always APPEND — NEVER overwrite existing content.

### 4.1 — Locate the daily log

1. Build path using the vault's `daily/` folder (identified via RAG in Step 1, or default `daily/`)
2. Build name: `<daily_folder>/YYYY-MM-DD.md` (using the meeting date)
3. If the file **exists**: open for append
4. If the file **does not exist**: create with the minimum structure below, then append:
   ```markdown
   ---
   date: "YYYY-MM-DD"
   tags: [daily]
   ---

   # 📅 YYYY-MM-DD

   ```

### 4.2 — Add entry to the meetings section

1. Check whether the section `## 🗓️ Meetings` exists in the file
2. If it **does not exist**: add at the end of the file:
   ```markdown

   ## 🗓️ Meetings
   ```
3. **Append** within the section (always at the end of the file, after the section header if just created):
   ```
   - [[YYYY-MM-DD_HH-MM_Title]] — <1-line summary> | Participants: X people
   ```
   - `[[YYYY-MM-DD_HH-MM_Title]]` = exact WikiLink to the meeting note file created in Step 3
   - 1-line summary = main objective of the meeting (max. 80 characters)
   - X = number of participants from frontmatter

### 4.3 — Append rules

- Use append write mode (`>>` or equivalent) — NEVER rewrite the file
- Do not duplicate entries: check whether a link to this note already exists before inserting
- Preserve all existing formatting of the daily log

---

## Step 5 — Save and Index

### 5.1 — Save the meeting note

1. Build file name: `YYYY-MM-DD_HH-MM_<Sanitized_Title>.md`
   - Sanitization: remove special characters, replace spaces with `_`
2. Save in the vault's meetings folder (identified by RAG in Step 1 as the folder containing meeting notes,
   or default `Meetings/`)
3. If the folder does not exist, create it before saving

### 5.2 — Save Confirmation
Log the successful creation of the file internally. Do NOT explicitly call indexing tools unless specifically requested by the environment, as this may consume unnecessary context.
